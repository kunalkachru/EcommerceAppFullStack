## Important Caveat

Android product behavior is in better shape than some older Android automation scripts suggest.

Root cause:
- some Android helper scripts are stale against the latest UI
- `scripts/e2e-adb.mjs` uses a paste-style key event in `clearAndType`, which can pollute typed values with clipboard content and create false failures

Impact:
- do not treat all old Android harness failures as product regressions
- the product itself was validated directly in the live emulator UI

## Automation Findings And Root Causes

This section is meant to separate real product risk from automation risk.

### Android automation findings

#### 1. Old Android flows still target older UI assumptions

Observed symptoms:
- older scripts looked for legacy navigation/content such as older home/product entry points
- some scripted failures reported missing category chips, missing similar strip, or incomplete home visual-search UI

Root cause:
- those scripts are partially out of sync with the current Home, Products, and ML layouts
- they are not always using the latest test IDs / resource IDs now present in the app

Conclusion:
- these failures are not strong evidence of broken product behavior by themselves
- live UI verification on the emulator showed the product flows working

#### 2. Android text entry helper can create false failures

Observed symptom:
- typed values could become polluted with clipboard text or unexpected prefixes
- this can corrupt API-key entry and ML query entry

Root cause:
- `scripts/e2e-adb.mjs` uses a paste-style key event inside `clearAndType`
- that behavior is unsafe for deterministic automation and can insert clipboard content before intended text

Conclusion:
- this helper should be fixed before relying on it for pass/fail decisions
- this is an automation defect, not a search-framework defect

#### 3. Android photo picker is hard to drive by filename alone

Observed symptom:
- the modern Android photo picker did not reliably expose stable filenames in the UI hierarchy during selection

Root cause:
- picker surface exposed clickable image cells and generic photo metadata instead of deterministic visible catalog filenames in the dumped hierarchy

Conclusion:
- image selection should use either:
  - stronger picker-aware automation, or
  - a seeded gallery flow with more deterministic hooks, or
  - image-cell targeting plus post-selection result assertions

#### 4. Android ML test false failures can come from provider/key mismatch

Observed symptom:
- one ML automation attempt reached the reasoning flow but failed with `401` from the provider

Root cause:
- the test harness temporarily paired the wrong provider selection with the available API key during debugging

Conclusion:
- this was a harness setup error, not a product retrieval failure

#### 5. Android should be considered functionally green, but automation needs hardening

Final Android conclusion:
- commerce flow: validated live
- baseline text retrieval: validated live
- LLM reasoning retrieval: validated live
- photo search: validated live
- remaining risk is primarily in harness reliability, not in confirmed product breakage

### iOS automation findings

#### 1. iOS flow is generally stable with the current Maestro setup

Observed status:
- login-to-orders flow passed
- ML flow passed

Conclusion:
- iOS automation is in materially better shape than the older Android helper path

#### 2. Save Password prompt is an intermittent UI blocker, but currently handled

Observed symptom:
- system `Save Password?` dialog can appear and block flows

Root cause:
- simulator/system modal interrupts the app flow

Current mitigation:
- shared dismissal flow is already wired into the Maestro scripts

Conclusion:
- keep this dismissal step in all critical end-to-end flows
- if it regresses, the failure is often environmental/modal, not core app logic

#### 3. iOS image-search assertion needs tolerant timing

Observed symptom:
- immediate assertion for `Closest match` warned once during ML flow
- the same run later successfully asserted `Best matches`

Root cause:
- result rendering timing is slightly variable after gallery selection

Conclusion:
- iOS ML image search is working
- the harness should assert with tolerant timing or with secondary result markers such as `Best matches`

### Cross-platform automation conclusion

- The current ML/search product behavior is stronger than the weakest automation scripts indicate
- The biggest remaining engineering gap is automation robustness, especially on Android
- The next agent should prioritize making the automation trustworthy before using it as the main regression signal
